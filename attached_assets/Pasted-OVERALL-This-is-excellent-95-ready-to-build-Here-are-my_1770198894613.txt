🔧 REFINEMENTS NEEDED:
1. FORMATION_STAGES - Add Stage Type
Problem: Not all stages are sequential. Some patterns have parallel paths.
Add column:
sqlstage_type: text NOT NULL  -- "sequential", "parallel", "terminal"
Example - VCP:

After T3, you can go to either:

T4+ (continues contracting) = sequential
Pivot Ready (done contracting) = sequential
Failed (breakdown) = terminal



Example - Cup and Handle:

Cup Complete → could branch to:

Handle Forming (normal)
Failed (breaks down before handle)



Why: AI needs to understand that some stages can skip or branch.

2. SETUP_VARIANTS - Add Required Criteria
Problem: Not all 25 criteria apply to all variants.
Example:

ORB doesn't care about "cup depth"
Multi-year cup doesn't care about "5-min relative volume"

Add column:
sqlrequired_criteria_ids: integer[]  -- Array of criteria IDs that MUST be scored
Alternative approach: Add to rating_criteria:
sqlapplicable_variants: integer[]  -- Null = applies to all, else specific variants
Why: Don't force AI to score irrelevant criteria. Focus on what matters per pattern.

3. RATING_CRITERIA - Split Max Points
Current: Every criterion = 4 points max
Problem: Not all criteria are equally important even within a category.
Example - Market Environment:

SPY trend alignment = Critical (4 pts)
Distribution day count = Important but secondary (3 pts)
Breadth = Nice to have (2 pts)

Suggested change:
sqlmax_points: integer NOT NULL  -- Can be 1-5, not always 4
Then adjust so each category still totals 20:

Market Environment: 5+4+4+4+3 = 20 pts
Volume Quality: 5+5+4+3+3 = 20 pts

Why: Captures nuanced importance within categories.

4. RATED_EXAMPLES - Add Context Fields
Missing critical context for AI learning:
Add columns:
sqlmarket_phase: text           -- "bull", "correction", "bear", "choppy"
sector_performance: text     -- "leading", "inline", "lagging"

                             --   "days_in_base": 42

Why: AI needs to learn that a "4-rated setup" in a bear market means something different than in a bull market. Context is everything.

5. RATING_WEIGHTS - Default vs Override
Your question 4: "Should rating_weights have a default set that gets copied per user, or be global?"
Answer: Both.
Suggested structure:
sqlrating_weights
  id: serial PRIMARY KEY
  setup_variant_id: integer REFERENCES setup_variants
  criteria_id: integer REFERENCES rating_criteria
  default_weight: decimal DEFAULT 1.0  -- System default
  user_id: integer                     -- NULL = global default
  weight: decimal                      -- User override
Logic:

On first run, system uses default_weight for all users
User can override specific weights (e.g., "I care more about volume")
User overrides stored with user_id, others use defaults
This allows personalization while maintaining baseline


6. FORMATION_STAGES - Add Duration Expectations
Add columns:
sqltypical_duration_min: text   -- "1 day", "1 week", "1 month"
typical_duration_max: text   -- "3 days", "3 weeks", "6 months"
too_long_threshold: text     -- When stage becomes stale
Example:

VCP T3 stage: typical = 1-3 weeks, too long = 8+ weeks (likely failed)
Cup forming: typical = 3-9 months, too long = 24+ months (dead)

Why: AI can flag "this pattern is taking too long, probably stale" and auto-lower score.
Answers your question 6: Yes, track time in stage.
